# D-4 scope — retiring work_lock from the ordering-dependent surface

Written at the close of D-3 (the 67-helper flip + TLS parkers). Read with
`model.md` (event flow) and the TLA models: `ProcControlLocks.tla` Phase C
proves the target protocol, `ProcControlLifetime.tla` the lifetime rules.

## What's left holding MTLock (inventory, post-D-3)

~87 sites: 46 in `process.C`, 38 in `processplat.C`, 2 in `handler.C`
(Windows-path re-locks inside `HandleLibrary`), 1 in `irpc.C`
(`IRPC::continueStoppedIRPC`), plus the handler batch bracket itself
(`MTManager` `startWork`/`endWork`).

Categorized:

1. **Ordering-free residue** (missed by option (b), same mechanics as the
   D-3 flip):
   - `ThreadPool` iterators + `size`/`find`/`getProcess` (~14 sites).
     Per-op locking today (work_lock is NOT held across `begin()`→`++`), so
     per-op proc_lock preserves semantics exactly.
   - `LibraryPool` readers (`size`, `getLibraryByName`, `getExecutable`,
     ~5 sites) — pool has `proc`, route exists.
   - `Library` readers (`getName`, `getLoadAddress`, ..., ~5 sites) —
     **open question**: `int_library` has no proc backpointer; either grow a
     weak proc route or argue immutability-after-load (name/addresses are
     decode-stamped; mutator audit needed).
   - `processplat.C` getters/setters that neither park nor arm the state
     machine (the `setDefault*` statics are global — Phase B).
2. **The D-2'd state-machine mutators** — already hold proc_lock alongside
   work_lock: `stopThread`/`continueThread`, `set/getSingleStepMode`,
   `set/getSyscallMode`, `postIRPC` (Process + Thread), `addBreakpoint`/
   `rmBreakpoint`, `runIRPCAsync`.
3. **Parking composites** — validate → arm → park: `runIRPCSync`,
   `addLibrary` (inject iRPC), `Process::stopProc`/`continueProc` (park in
   `waitAndHandleForProc`), `IRPC::continueStoppedIRPC`, and the
   `processplat.C` methods that wait on responses (`setTrackLibraries`,
   `refreshLibraries`, `refreshThreads`, ...).
4. **The handler batch bracket** — `startWork`/`endWork` around each
   `waitAndHandleEvents` drain in `MTManager::evhandler_main`, plus the
   self-handling modes (`NoThreads`/`GeneratorThreading`) where the USER
   thread runs the same drain under `MTLock(allow_init)`.
5. **Phase B surface (NOT D-4):** `handleEvents`, `registerEventCallback`
   ×4, `setThreadingMode`, `createProcess`/`attachProcess` bootstrap,
   the `setDefault*` globals, cb_lock formalization.

## Why the D-2'd mutators can now drop work_lock

The S5 post-mortem (CLAUDE.md Status): the handler asserts that killed the
full removal were missing mutex+visibility between user-side state-machine
mutators and the handler's `syncRunState` — NOT a fundamental ordering
problem. The ordering (arm → SIGSTOP → decode → handle) is the kernel
round-trip. Exclusion is now proc_lock on both sides: handler via S4
`handle_plock`, user via D-2. The Phase C TLA model proves the target
protocol deadlock- and race-free (`AtMostOneWriterPerProc`, BUG10 knob =
handler without proc_lock).

What work_lock STILL provides these methods today, and its replacement:

| work_lock service | replacement |
|---|---|
| user-mutator vs handler-batch exclusion | proc_lock (both sides, done) |
| user-mutator vs user-mutator (2 threads, same proc) | proc_lock |
| user ops on DIFFERENT procs serialized | deliberately dropped (the point) |
| visibility of mode flags into syncRunState | proc_lock acq/rel pairing |
| cross-proc fork handling order | lock BOTH procs ascending (BUG9 knob) |

## Increments (each gated validate_pw.sh + TSan sweep, committed separately)

- **D-4a — ordering-free residue** (category 1): ThreadPool iterator family
  first (pure mechanics), then LibraryPool, then Library (after the route /
  immutability question resolves), then processplat getters. Low risk.
  **LANDED** (each gated 1534/1534 + 0 TSan inversions): `26b98843d`
  ThreadPool (pool weak route; unlocked getProcess deref fixed),
  `031b95012` LibraryPool methods, `b374c15c9` Library readers (immutability
  FAILED — refresh_libraries mutates published libs; getDynamicAddress/
  isSharedLib pre-existing races closed; ~Library lock-free by refcount),
  `62395e2e8` processplat direct-impl 10 (named-ref restructure; TOCTOU +
  exited-NULL-deref closed).  Still open in D-4a: LibraryPool ITERATORS
  (never locked at all — route needed on the iterator objects), and the
  deferred parkers listed under D-4c.
- **D-4b — drop MTLock from the D-2'd mutators** (category 2): the flip for
  the state-machine surface. The S5 assert set (HandleThreadStop
  pending-stop, HandleBreakpointClear all-stopped, desync/restore chains,
  iRPC state asserts) are the tripwires — any assert in the gate = stop and
  re-diagnose, do not weaken asserts.
- **D-4c — parking composites** (category 3): TLS-parker pattern per method
  (validate+arm under proc_lock → release → park → re-validate + re-derive
  impl in every post-park bracket). `stopProc`/`continueProc` park via
  `waitAndHandleForProc`, which in self-handling modes RUNS the handler on
  the user thread — coordinate with D-4d.
- **D-4d — the handler flip** (category 4, LAST): delete
  `startWork`/`endWork`; the handler holds only `handle_plock` per event
  (+ Counters, which have their own locks). Self-handling modes make the
  user thread take the same per-event proc_lock path. The two `handler.C`
  Windows blocks become locked-accessor scopes. After D-4d, MTLock's only
  users are the Phase B surface.

Rationale for the order: a→c empty the user-side surface while the handler
batch keeps its global bracket as a safety net; d is the single riskiest
step and goes last, with the TLA-proven protocol, the tripwire asserts, and
TSan as the nets.
